## 3. 구현 흐름 (`.pc`)

```
W1257S01()                          # 진입 dispatch
 ├─ prthdr / getmid / getmod        # COMHDR + mid/mod 주소 획득
 ├─ e_msg 기본설정 (KR / S / M)
 ├─ switch(mid->prGb[0])
 │    case '1' → W1257S01_F1()      # 조회
 │    default  → comhdr_o.mesg "작업구분 오류." 후 즉시 return(sndl)
 ├─ retc==RET_SUCCESS → eflg="0" + l_dbcommit()
 │            else      → eflg="1" + l_dbrollback()
 ├─ fnSetMsg(&e_msg, comhdr_o)      # e_msg.code → 응답 메시지 세팅
 └─ return(content->sndl)

W1257S01_F1()                       # 조회 본체
 ├─ getmid / getmod, sndl += W1257S01_O_SZ, mod 0-clear
 ├─ max_row = MAX_RRB / sizeof(orec[0])
 ├─ 페이징 키(vKeyv) 결정
 │    K_NEXT → content_get_nkey
 │    K_PREV → content_get_pkey
 │    default(최초) → "99991231999999"
 ├─ 입력 유효성: atoi(frRgsnYmd) > atoi(toRgsnYmd) → e_msg 10781, RET_FAILURE
 ├─ DECLARE/OPEN  W1257S01_C1 (TB_FXB_CMC008L)
 │    OPEN 오류 → e_msg 10118 (return 없음)
 │    NO_DATA   → e_msg 91001
 ├─ for(;;) FETCH W1257S01_C1
 │    FETCH 오류(≠NO_DATA) → CLOSE + e_msg 10118 + RET_FAILURE
 │    NO_DATA → break
 │    ii==max_row → i_isnext++; break   # 다음 페이지 존재 sentinel
 │    zsedit 로 orec[ii] 채움 + vKeyv = cKeyv (NEXT KEY)
 ├─ CLOSE W1257S01_C1
 ├─ 결과 판정
 │    ii==0 && i_isnext==0 → e_msg 91001 (자료 없음)
 │    i_isnext==1          → content_set_nkey + e_msg 91002 (계속됩니다)
 │    else                 → e_msg 91003 (완료)
 ├─ zredit(grid01_cnt, ii)
 ├─ sndl += (ii-1) * sizeof(orec[0])
 └─ return(RET_SUCCESS)
```

트랜잭션: 조회 전용이나 컨벤션상 성공 시 `l_dbcommit()`, 실패 시 `l_dbrollback()` 명시 호출.

## 5. 로직 / 필터 규칙

- **작업구분 분기**: `prGb='1'` 만 조회. 그 외는 "작업구분 오류." 반환 (커서 미실행).
- **커서 `W1257S01_C1`** (`TB_FXB_CMC008L`):
  - 기간 필터: `RGSN_YMD BETWEEN :vFrRgsnYmd AND :vToRgsnYmd`.
  - 대상 필터 (`PBNS_TGT_DCD` 공지대상구분):
    - `'0'` (개인): `PBNT_TGT_ID = NVL(:vUserId, PBNT_TGT_ID)` — userId 미입력 시 전체.
    - `'1'` (법인/전행): `PBNT_TGT_ID = NVL(:vCompanyCd, PBNT_TGT_ID)` — companyCd 미입력 시 전체.
    - `'2'` (공통): 무조건 포함.
  - 페이징 필터: `RGSN_YMD || PBNT_RGSN_SQN < :vKeyv`, `ORDER BY RGSN_YMD DESC, PBNT_RGSN_SQN DESC` (최신순).
  - `@URL:` 파싱: `PBNS_CON` 에 `@URL:` 마커가 있으면 앞부분=본문(`F_PBNS_CON`), 뒤부분=URL(`L_PBNS_CON`) 로 분리.
- **페이징**: 최초 조회는 `vKeyv="99991231999999"` (전체 상위). K_NEXT 는 저장된 nkey 로 이어감. `max_row` 초과 fetch 시 `i_isnext=1` 로 다음 페이지 존재 표시하고 마지막 1건은 저장하지 않음.
- **입력 검증**: 시작일 > 종료일 이면 `10781` ([조회일자] 종료일자 이전의 시작일자 입력) 후 RET_FAILURE.
- **에러/메시지 (e_msg.code → mesg, eflg)**:
  | code | 의미 | 발생 |
  |-----|-----|-----|
  | `10781` | 종료일자 이전 시작일자 입력 | frRgsnYmd > toRgsnYmd |
  | `10118` | 자료 조회중 오류 | 커서 OPEN/FETCH 오류 |
  | `91001` | 해당하는 자료가 없습니다 | OPEN NO_DATA / 반환 0건 |
  | `91002` | 조회가 계속됩니다 | 다음 페이지 존재 (nkey set) |
  | `91003` | 조회가 완료되었습니다 | 마지막 페이지 |
  - eflg: 성공 `"0"` (commit) / 실패 `"1"` (rollback).

## 6. 주의 — 문서/코드 불일치

| 위치 | 헤더/주석 표기 | 실제(.pc) |
|-----|-----|-----|
| `.h` `frRgsnYmd`/`toRgsnYmd` | "등록시각 F / 등록시각 T" | 등록**년월일**(`RGSN_YMD`) 로 BETWEEN 비교. `.pc` 내부 변수/로그도 "등록년월일 F/T". 헤더 주석 오기. |
| `.pc:169-182` K_PREV 페이징 | "이전" 분기로 `content_get_pkey` 사용 | 커서 조건은 항상 `... < :vKeyv` + `ORDER BY ... DESC` 로 NEXT 와 동일 방향. 역방향(이전 페이지) 비교/정렬이 없어 K_PREV 가 실질적으로 이전 페이지를 되돌리지 못함 (페이징 방향 버그 소지). |
| `.pc:245-249` 커서 OPEN 오류 | — | OPEN 오류(`10118`) 시 로그+code 세팅만 하고 `return` 하지 않아 이어지는 FETCH 루프로 진입. 최종적으로 FETCH 오류에서 RET_FAILURE 반환에 의존. |
| `.pc:251-255` OPEN 후 NO_DATA | — | 커서 OPEN 은 통상 NO_DATA_FOUND 를 반환하지 않음 — 사실상 도달하지 않는 방어 코드. 실 무자료 판정은 하단 `ii==0` 분기(91001)가 담당. |

| `.pc:47-48,124` `mod`(W1257S01) | 주석 모두 "입력값 주소" | `mod` 는 출력값 주소. 복붙 주석 오기. |
