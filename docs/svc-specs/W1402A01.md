# 서비스 명세 — `W1402A01` (고객정보 수정)

## 3. 구현 흐름 (`.pc`)

```
W1402A01()                      ← 진입 (콜백, 함수 주석 "조회"는 오기)
  ├─ prthdr / getmid / getmod   COMHDR + input(mid) + output(mod) 준비
  ├─ E_MSG 기본설정              ltyp="KR", mtyp="S"(FXON), rtyp="M"(Message전송)
  ├─ switch(mid->prGb)
  │     '2' → retc = W1402A01_F2(...)     ← 실제 로직 (수정)
  │     그외 → comhdr_o->mesg="작업구분 오류.", eflg="1",
  │            e_msg.code="32134" 세팅 후 즉시 return
  └─ retc==RET_SUCCESS ? l_dbcommit() : l_dbrollback()   (명시적 커밋, AUTOCOMMIT 아님)

W1402A01_F2()                   ← 사용자정보 수정
  ├─ getmid/getmod, memset(mod)
  ├─ 입력 바인딩: XPRINTF + l_rtrim
  │     fxUserNo/cpn1/userBlngDvm/userJcm/emlId1/emlDmnNm1/sndgMdiaDcd/acnPwdLV
  │     hls_upd_id ← comhdr_i->usid (최종변경 ID)
  ├─ 입력 유효성 체크(핸드폰/Email 필수)는 #if 0 로 비활성 — 화면단 검증 위임 (§6)
  ├─ if (strlen(vacnPwdLV) > 0)                ← 비밀번호레벨 수정 분기
  │     EXEC SQL UPDATE TB_FXB_CSC004M
  │       SET ACN_PWD_SECU_LVL=:vacnPwdLV, FX_SYS_LSMD_ID=:hls_upd_id, FX_SYS_LSMD_TS=SYSDATE
  │       WHERE FX_USER_NO=:fxUserNo
  │   else                                     ← 연락처/Email 수정 분기
  │     EXEC SQL UPDATE TB_FXB_CSC004M
  │       SET USER_CPN, USER_BLNG_DVM, USER_JCM, USER_EML_ID, USER_EML_DMN_NM,
  │           SNDG_MDIA_DCD, FX_SYS_LSMD_ID=:hls_upd_id, FX_SYS_LSMD_TS=SYSDATE
  │       WHERE FX_USER_NO=:fxUserNo
  ├─ SQLCODE 처리 (두 분기 동일)
  │     NO_DATA_FOUND → e_msg "91001"(해당 자료 없음), return SQL_SUCCESS  ← §6: 성공 취급
  │     그외 오류      → e_msg "10422"(수정 중 오류), return RET_FAILURE   → rollback
  └─ 정상 → zredit(mod->grid01_cnt,1), e_msg "10150"(정상처리), return RET_SUCCESS → commit
```

## 4. 사용 DB 테이블

| 테이블 | 한글명 | 별칭 | 접근 | 용도 |
|---|---|---|---|---|
| `TB_FXB_CSC004M` | 사용자정보기본 | — | **UPDATE** | `FX_USER_NO` 키로 사용자 1행 수정. 분기에 따라 계좌비밀번호레벨(`ACN_PWD_SECU_LVL`) 또는 연락처/팀명/직명/Email/발송매체(`USER_CPN`·`USER_BLNG_DVM`·`USER_JCM`·`USER_EML_ID`·`USER_EML_DMN_NM`·`SNDG_MDIA_DCD`) 를 갱신. 두 분기 공통으로 `FX_SYS_LSMD_ID`(최종변경ID)·`FX_SYS_LSMD_TS`(SYSDATE) 스탬프 |

*DML 요약: `TB_FXB_CSC004M` 에 UPDATE 1건 (분기별 SET 절만 상이, SELECT/INSERT/DELETE 없음). 공통함수 경유 간접 테이블 접근 없음.*

## 5. 로직 / 필터 규칙

- **작업구분**: `prGb=='2'` 만 처리(수정). 나머지는 `32134` + "작업구분 오류." 로 즉시 거부.
- **분기 UPDATE (배타적)**: `strlen(vacnPwdLV) > 0` 여부로 두 UPDATE 중 하나만 실행.
  - `acnPwdLV` 값이 있으면 **계좌비밀번호레벨만** 갱신 (연락처/Email 필드는 손대지 않음).
  - 없으면 **연락처/팀명/직명/Email/발송매체** 를 갱신 (비밀번호레벨은 손대지 않음).
  - → 한 번의 호출로 두 그룹을 동시 수정할 수 없음. 호출측이 목적에 맞게 `acnPwdLV` 세팅 여부를 제어해야 함.
- **입력 유효성**: 핸드폰/Email(ID/도메인) 필수 검사 로직은 `#if 0` 로 주석 처리(비활성). "화면에서 발송매체별로 입력값 체크" 하므로 서버 검증을 뺐다는 코드 주석. 서버는 사실상 빈 값도 그대로 UPDATE 함.
- **트랜잭션**: `RET_SUCCESS→l_dbcommit / 실패→l_dbrollback`. AUTOCOMMIT 아님 — 명시적 커밋 필수.
- **에러 (e_msg.code→mesg, eflg)**:
  - `10150` 정상처리 되었습니다(조회로 확인) — 성공.
  - `91001` 해당하는 자료가 없습니다 — 대상 행 미존재(`NO_DATA_FOUND`). **주의**: `return SQL_SUCCESS` 라 상위에서 성공 취급 → `eflg="0"` + commit (§6).
  - `10422` 사용자정보 수정중 오류 발생 — SQL 오류 → `RET_FAILURE` → rollback.
  - `32134` 작업구분 오류 (진입부 default).
- **결과 전달**: 데이터 대신 COMHDR — `eflg`(0 정상/1 오류) + `e_msg.code`→`mesg`. WTG `/v1/tx` 는 `errn`≠0 이어도 전문 있으면 200 + COMHDR 본문 동봉(레거시 규약).

## 6. 주의 — 문서/코드 불일치 (마이그레이션 시 확인)

| 위치 | 헤더/주석 표기 | 실제(`.pc`) |
|---|---|---|
| `W1402A01()` 함수 주석 | "조회" | 수정 처리 dispatch (UPDATE) |
| `.pc` `mod` 변수 주석 (49,126행) | "입력값 주소" | 출력값 주소 (복붙 오기) |
| `.h`/`.pc` `acnPwdLV` 주석 | "주문비밀번호체크단계" | 계좌비밀번호레벨(`ACN_PWD_SECU_LVL`, SQL 주석 = "계좌비밀번호레벨") |
| `.h` `grid01_cnt` 주석 | "그리드건수" | 실제 grid 없음. 성공 시 상수 `1` 만 세팅되는 처리성공 플래그 |
| default case `e_msg.code` 주석 | "등록중 오류 발생" (`32134`) | 실제 상황은 "작업구분 오류" — 코드 텍스트와 상황 설명 불일치 |
| `NO_DATA_FOUND` 처리 | (수정 실패로 보이나) | `91001` 메시지 세팅 후 `return SQL_SUCCESS` → **상위에서 성공(commit)으로 취급**. 대상 사용자 없어도 오류가 아닌 정상 응답으로 나감 |
| 입력 필수 검증 (핸드폰/Email) | 헤더상 값 존재 전제 | `#if 0` 로 **비활성** — 서버는 빈 값도 그대로 UPDATE |

> WTG 는 `.h` 를 파싱해 필드명/크기만 취하고 전문을 통과시키므로 위 주석 오기가
> 동작에 영향 없다. 단 `acnPwdLV` 값 유무가 두 UPDATE 를 가르는 배타 분기라는 점,
> 그리고 대상 미존재(`91001`) 가 성공으로 응답된다는 점은 클라이언트/운영 관점에서
> 반드시 인지해야 한다.
