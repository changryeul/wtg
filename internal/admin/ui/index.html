마진 테이블

5-layer 마진 + 시간대를 담은 단일 JSON. 저장 시 모든 mci-price 가 atomic 으로 새 테이블로 전환. Version 은 운영자가 손으로 1씩 올리는 게 좋습니다 (감사용).

📘 schema 안내 (5-layer + time window)
hq_margin: 본점 마진 (pair × tier). Tier="" = 와일드카드.
site_margin: 영업점·채널 마진 (pair × channel × site).
customer_margin [신규]: 고객별 추가/대체 마진. mode="add" (default) | "override". priority 큰 값 우선.
skew_amount / spread_amount [P6]: bid/ask 동방향 shift (skew) / 양쪽 폭 추가 확대 (spread). 모든 마진 카테고리에 적용. 0=기존 동작.
swap_point: 만기별 forward 마진 (pair × tenor).
time_windows [신규]: 시간대 정의. 각 margin entry 의 window 필드가 참조.
· 모든 entry 의 window 는 optional — 비면 모든 시간대 적용.
· customer_margin.mode="override" 는 HQ+Site 무시하고 customer 마진 단독.
· time_windows[].complement_of 로 보집합 정의 (예: off_hours = regular 반대).

gRPC 구독자

            
            
            
          

          

            이 페이지가 무엇을 보여주는가?
            

              
📡 mci-price 서버 입장에서 어떤 internal service 가 시세 stream 을 받아가는지 보여줍니다. 사용자 (고객) 가 아님 — 사용자는 「🔗 연결 (ws)」 페이지에 보입니다.

              
    
                
  • Tick: 다중시장 BEST 산정 직후 raw 시세 (마진 미적용). 보통 edge-price 가 받아 ws fan-out.
    • 
                
  • Quote: Profile (예: WEB.BRANCH.VIP) 별 마진 적용된 가격. edge-price 가 명시 구독 시만.
    • 
                
  • Bar: 1초/1분/5분/15분/1시간/1일 봉 close 발사. mci-chart 가 받음.
    • 
                
  • Customer Quote: customer 별 별도 마진 (CustomerRule) 적용. Phase 4c (--customer-stream) 활성 인스턴스만.
    • 
              

              
큐 색상: 정상 / 50%+ 적체 / 80%+ 위험 — 80% 이상이면 backpressure WARN 로그도 발생합니다.

            

          

          

        


        
        

          

            
ws 연결

            
            
            필터:
            
            
            
            
            
          

          

            이 페이지가 무엇을 보여주는가?
            

              
🔗 mci-edge-price 입장에서 어떤 외부 사용자가 ws 로 붙어있는지. 사용자 본인이 여기 보입니다 (admin UI 「📡 WS 모니터」 / 「💱 시세 라이브 ws」 페이지를 열어두면 그 ws 도 같이 잡힙니다).

              
    
                
  • profile_key: 어떤 마진이 적용될지 결정 (Channel.Site.Tier). 빈값이면 quote 미수신 (raw 만).
    • 
                
  • customer_id: Phase 4c (--customer-stream) 활성 시 채워짐. 빈값이면 Profile-only 모드.
    • 
                
  • pairs: 사용자가 명시 구독한 통화쌍. null 이면 모든 통화쌍 수신.
    • 
                
  • queue: edge-price → 그 ws 클라이언트로 송신 대기열. 사용자 측이 느리면 적체.
    • 
                
  • by_profile: 전체 분포 (필터 무관). 컨텍스트 유지용.
    • 
              

              
필터 사용: customer_id 입력하면 그 사용자만 / profile 입력하면 그 등급만.

            

          

          

          

        


        
        

          

            
Customer 검색

            
            
            
          

          

            이 페이지가 무엇을 보여주는가?
            

              
🔍 support 시나리오 — 티켓에 customer_id 받았을 때 mci-price 등록 + edge ws 연결 을 한 번에 확인합니다. 양 layer 를 따로 호출할 필요 없음.

              
    
                
  • ✓ mci-price 등록 + edge 연결 1개+ → 정상
    • 
                
  • ✓ mci-price 등록 + edge 연결 0개 → 사용자가 ws 끊김 (브라우저 닫음)
    • 
                
  • ✗ mci-price 미등록 + edge 연결 1개+ → edge 의 RegisterCustomer 실패 (Phase 4c 비활성?)
    • 
                
  • ✗ 둘 다 0개 → 한 번도 연결 안 했거나 존재 안 함
    • 
              

              
Phase 4c (--customer-stream) 가 활성이어야 customer_id 가 채워집니다. 비활성이면 모든 customer 가 "" 라 검색 무의미.

            

          

          

        


        
        

          

            
Backpressure 이력

            
            
            
          

          

            이 페이지가 무엇을 보여주는가?
            

              
⚠️ 큐 점유율이 80% 도달 시 (drop 직전 단계) 발사되는 WARN 의 누적 + 최근 이력 (ring buffer 100건). 1분에 키당 1회 발사.

              
    
                
  • mci-price: 4 fan-out path (tick/quote/bar/customer_quote). srv_id 가 어느 edge 인스턴스인지 알려줌.
    • 
                
  • mci-edge-price: ws Subscriber.Send 단계. profile 라벨 (어느 Profile 사용자가 정체).
    • 
              

              
운영 흐름: slow consumer 격리 알람 본 후 본 페이지에서 "그 직전 어디서 정체 누적됐나" 확인 → 「🔌 구독자」/「🔗 연결」 페이지에서 현재 상태 cross-check.

            

          

          

          

        


        
        

          

            
FX swap 잠금 통계

            
            
            
          

          

            이 페이지가 무엇을 보여주는가?
            

              
🔁 매칭 엔진이 호출하는 POST /v1/quote/swap/lock (FX swap = near leg + far leg 동시 잠금) 의 누적 카운터.

              
    
                
  • requests / successes: 차이가 누적되면 운영 alert. 보통 ≤ 0.1% 가 정상.
    • 
                
  • fail_near / fail_far / fail_swap_index: 어느 단계에서 부분실패했는지. near→far→swap_index 순차 시도 — 뒤로 갈수록 위험도 ↑ (다른 leg revoke 필요).
    • 
                
  • revoke_ok / revoke_fail: 부분실패 시 best-effort revoke 결과. fail 증가 시 stale quote_id 가 Registry 에 잔존 — 매매 AP 측 validate 가 reject 처리해야.
    • 
              

              
⚠️ 운영 alert 기준 (suggested): fail_far + fail_swap_index > requests × 0.1% → Registry 또는 SwapIndex (Redis) 점검. revoke_fail > 0 → 즉시 분쟁 큐 (leg 잔존 위험).

              
상세: docs/swap-trade-spec.md §4.5.

            

          

          

          

          

        


        
        

          

            
Algo Stream (SubscribeAlgo)

            
            
            
          

          

            이 페이지가 무엇을 보여주는가?
            

              
🤖 내부 algo 봇 전용 시세 stream 의 실시간 카운터. edge-price ws (외부 고객용) 과 격리된 gRPC 채널.

              
    
                
  • subscribers_active: 지금 접속한 algo 클라이언트 수. 예상보다 낮으면 봇이 재접속 중이거나 crash.
    • 
                
  • ticks_emitted: fan-out 된 live tick 누적. rate 로 실제 활동 판단.
    • 
                
  • send_drops: per-client buffer 폭주 drop. 지속 증가 = slow client 존재.
    • 
                
  • backfill_emitted: 재접속 시 replay 한 tick. 정상 재접속 흐름.
    • 
                
  • backfill_gaps: ring 밖 요청 → snapshot 재구독 유도한 세션 수. 지속 증가 = ring_size 부족.
    • 
                
  • disconnected_slow: slow client timeout evict 누적. 정상은 0. 발생 시 client 로직 점검.
    • 
                
  • symbols_with_ring: 지금까지 tick 이 흐른 심볼 수. ring_size 는 각 심볼별 capacity.
    • 
              

              
운영 tip: --algo-ring-size 는 최대 disconnect 견딜 시간에 맞춰. 초당 100 tick/심볼 × 5분 = 30k 정도.

              
Prometheus metric: wtg_algo_* (curl /metrics 로 확인).

            

          

          

          

        


        
        

          

            
quote-forwarder 통계

            
            
            
          

          

            이 페이지가 무엇을 보여주는가?
            

              
📡 quote-forwarder 가 UDP 시세 → broker (또는 mci-price gRPC) 로 forward 하는 통계.

              
    
                
  • received: UDP 로 받은 누적 envelope. cooker (시세 source) 가 publish 한 양.
    • 
                
  • published: 정상 forward 한 누적. received - published ≈ reject 합.
    • 
                
  • invalid_quotes: fastExtractV1 가 reject 한 envelope 의 reason 별 분류.
                  
      
                    
    • non_positive_price: bid 또는 ask 가 음수/0. cooker drift 또는 scale 오류 alert.
      • 
                    
    • crossed_spread: ask < bid. cooker 의 spread 계산 오류.
      • 
                    
    • not_a_quote: 35=X trade message 등 — silent skip 의도된 동작.
      • 
                    
    • missing_symbol: 55= 누락. FIX 메시지 깨짐.
      • 
                  
    
                
    • 
              

              
운영 alert: non_positive_price > 0 또는 publish_errors 가 빠르게 증가 시 cooker / broker 진단 필요.
