## 3. 구현 흐름 (`.pc`)

```
W1113S03()                          // 콜백 진입
  prthdr / getmid / getmod          // 공통헤드 + I/O 주소 획득
  e_msg 기본설정 (KR/S/M), comhdr_o->rtyp="M"
  switch(mid->prGb[0])
    '1' | '2' | '3' → W1113S03_F1() // 세 케이스 모두 동일 (rev 1.01 주석과 불일치, §6)
    default         → "작업구분 오류." 즉시 return(sndl)
  retc == RET_SUCCESS → l_dbcommit()   // 조회만 하지만 명시적 commit
  retc != RET_SUCCESS → l_dbrollback()
  return(content->sndl)

W1113S03_F1()                       // PU리스트 조회
  content->sndl += W1113S03_O_SZ; memset(mod)
  max_row = MAX_RRB / sizeof(orec[0])
  content->keyc → vKeyv (K_NEXT: nkey / K_PREV: pkey / 최초: "0")   // 페이징 키
  yprintf → vPuCd / vFxUserNo / vPuPtrnDcd (+ l_rtrim)
  DECLARE W1113S03_C1 CURSOR (SELECT ... TB_FXB_CMG026M)
    SQLCODE!=0 → e_msg 10118, eflg="1" (return 없음, 계속 진행 §6)
  OPEN C1
    NO_DATA_FOUND → e_msg 91001, eflg="0", return RET_FAILURE
    기타 err      → e_msg 10118, eflg="1", return RET_FAILURE
  for(ii=0;;ii++)
    FETCH C1 INTO cPuCd,cPuNm,cUseYn,cPuPtrnDcd
      NO_DATA_FOUND → break
      기타 err      → CLOSE + e_msg 10118 eflg="1" return RET_FAILURE
    if(ii==max_row) i_isnext++; break;    // 다음페이지 존재
    l_rtrim + zsedit → mod->orec[ii]
    sprintf(vKeyv,"%.64s",cKeyv)          // cKeyv 미할당 — NEXT KEY 버그 (§6)
  CLOSE C1
  분기:
    ii==0 && i_isnext==0 → e_msg 91001 (자료없음), eflg="0"
    i_isnext==1          → content_set_nkey, e_msg 91002 (계속)
    else                 → e_msg 91003 (완료), grid01_cnt=ii
  CLOSE C1                                // 중복 CLOSE (§6)
  eflg="0"; grid01_cnt=ii
  content->sndl += (ii-1)*sizeof(orec[0]) // ii==0 시 음수 가산 (§6)
  return RET_SUCCESS
```

## 4. 사용 DB 테이블

| 테이블 | 한글명 | 별칭 | 접근 | 용도 |
|-----|-----|-----|-----|-----|
| `TB_FXB_CMG026M` | FX북(PU) 마스터 | `CMG026M` | SELECT | FX북ID/명/사용여부/운영유형 목록 조회 |

*DML 요약: 단일 커서 `W1113S03_C1` SELECT 만. INSERT/UPDATE/DELETE 없음. 조회 전용임에도 콜백에서 `l_dbcommit()`/`l_dbrollback()` 명시 호출.*

SELECT 컬럼: `FXBK_ID, FXBK_NM, USE_YN, FXBK_OPRT_PTRN_DCD`
WHERE: `FXBK_ID LIKE NVL('%'||:vPuCd||'%', FXBK_ID) AND USE_YN='Y'`
ORDER BY: `FXBK_ID`

## 5. 로직 / 필터 규칙

- **필터**: 실제 유효 필터는 `puCd`(부분일치 LIKE)와 `USE_YN='Y'` 두 가지뿐. `NVL('%'||:vPuCd||'%', FXBK_ID)` 는 `vPuCd` 공백이어도 `'%%'` 로 NULL 이 아니라 NVL 이 무의미 — 사실상 항상 `LIKE '%<puCd>%'`. `puCd` 공백이면 전체 사용중 FX북 반환.
- **미사용 입력**: `fxUserNo`, `puPtrnDcd` 는 `yprintf`/`l_rtrim`/로그까지 처리되지만 커서 WHERE 에 바인딩되지 않음 → 실 조회에 영향 없음.
- **페이징**: `content->keyc`(K_NEXT/K_PREV/최초)로 `vKeyv` 결정하나, 커서 WHERE 에 `vKeyv` 바인딩이 없어 offset/seek 미적용. `ii==max_row` 도달 시 `i_isnext=1` 로 "계속" 표시하지만 NEXT KEY 로 넣는 `cKeyv` 가 미할당(§6)이라 실제 다음페이지 seek 불가.
- **트랜잭션**: 조회 전용. 콜백 레벨에서 F1 성공 시 `l_dbcommit()`, 실패 시 `l_dbrollback()`.
- **에러/메시지 코드** (`e_msg.code` → `fnSetMsg`, `comhdr_o->eflg`):
  | code | 의미 | eflg |
  |-----|-----|-----|
  | `10118` | 커서 DECLARE/OPEN/FETCH SQL 오류 | `"1"` |
  | `91001` | 해당 자료 없음 | `"0"` |
  | `91002` | 조회 계속(다음 페이지 존재) | `"0"` |
  | `91003` | 조회 완료 | `"0"` |
  - `rtyp="M"`(메시지 전송), `ltyp="KR"`, `mtyp="S"`(FXON).
  - 함수 말미 `eflg="0"` 를 무조건 재설정 → 정상 조회는 eflg 0. (10118 FETCH 오류는 그 전에 return 하므로 영향 없음)
