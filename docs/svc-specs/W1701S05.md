# 서비스 명세 — `W1701S05` (코드명 조회)

> 작성 양식: `docs/svc-spec-template.md`. 본 명세는 `.pc` 실 구현 기준이며,
> `.h` 헤더 주석과 어긋나는 부분은 §6 주의에 정리했다.

## 1. 개요

| 항목 | 값 |
|---|---|
| 서비스 코드 | `W1701S05` |
| 이름 | 코드명 조회 (`.pc` Description = "코드명 조회". `.h` 헤더는 "메뉴상세조회" — §6) |
| 유형 | **조회 (SELECT)** — 두 공통코드 그룹(FX사용자구분·화면접근권한구분)을 UNION ALL 로 전량 반환 |
| 구현 | `win/src/trn/W1700/W1701S05.pc` (277 lines) |
| I/O 헤더 | `win/src/inc/trn/W1701S05.h` |
| 라우팅 alias | `W1701S05` → `POST /v1/tx {"alias":"W1701S05","data":{…}}` |
| 채널 | 진입부에 `ctyp→chnl` 매핑 있음 (HTS/MTS/WTS/EMP/API/SPI → `chnl` 복사, 그 외 로그만) |

## 2. I/O 전문 (`.h`)

**Input `W1701S05_I`** (총 2B)

| 필드 | 크기 | 설명 (실제) |
|---|---|---|
| `fXuSerDcd` | 2 | 권한그룹코드 — `.h` 상 입력 필드. **`.pc` 에서 전혀 참조되지 않음** (쿼리에 바인딩 없음, §6) |

**Output `W1701S05_O`** — grid (orec 가변)

| 필드 | 크기 | 설명 |
|---|---|---|
| `grid01_cnt` | 6 | 그리드01 건수 — fetch 한 행 수(`ii`)를 `zredit` |
| `orec[].wKtp` | 1 | 코드구분 — 리터럴 `'1'`(FX사용자구분) / `'2'`(화면접근권한구분) |
| `orec[].code` | 2 | 코드 — `CMC003D.CMCD_VL`. 소스 커서 버퍼는 `[30+1]` 로 받고 2B 로 truncate (§6) |
| `orec[].codeNm` | 30 | 코드명 — `CMC003D.CMCD_VL_NM` |

> `orec[1]` 은 헤더상 1건 자리만 선언, 실제는 `content->sndl` 을 fetch 건수만큼 늘려 가변 grid 로 전송.

## 3. 구현 흐름 (`.pc`)

```
W1701S05()                        ← 진입 (콜백)
  ├─ getmid / getmod             COMHDR + input(mid) + output(mod) 준비
  ├─ E_MSG 기본설정              pname=rkey, ltyp=comhdr_i->ltyp, mtyp="S"(FXON), rtyp="M"
  ├─ switch(comhdr_i->ctyp[0])   접속매체구분 → comhdr_i->chnl 복사
  │     '1'HTS '2'MTS 'W'WTS 'E'EMP 'A'API 'S'SPI → chnl=ctyp
  │     그외 → "No Mapping" 로그만
  ├─ retc = W1701S05_F1(...)     ← 실제 조회 로직
  └─ retc==RET_SUCCESS ? l_dbcommit() : l_dbrollback()   (명시적 커밋, 순수 SELECT라 명목상)
     └─ fnSetMsg(&e_msg, comhdr_o) → return content->sndl

W1701S05_F1()                     ← 조회
  ├─ getmid/getmod, sndl += W1701S05_O_SZ, memset(mod)
  ├─ max_row = MAX_RRB / sizeof(orec[0]);  comhdr_o->rtyp = "G"(grid)
  ├─ GRID key 처리 (content->keyc)  K_NEXT→content_get_nkey / K_PREV→content_get_pkey / default→"0"
  │      → v_keyv 세팅 (계산만 하고 커서 WHERE 에 미사용 — §6 페이징 미동작)
  ├─ EXEC SQL DECLARE W1701S05_C1 CURSOR FOR
  │      SELECT '1', CMCD_VL, CMCD_VL_NM FROM TB_FXB_CMC003D WHERE COM_CD_NM='FX_USER_DCD'
  │      UNION ALL
  │      SELECT '2', CMCD_VL, CMCD_VL_NM FROM TB_FXB_CMC003D WHERE COM_CD_NM='SCRE_ACS_ATHR_DCD'
  │      (DECLARE 직후 SQLCODE!=0 검사 → e_msg "95000" set, return 은 주석처리 — §6)
  ├─ EXEC SQL OPEN C1
  │      SQLCODE!=0 → "95000" + RET_FAILURE
  │      else NO_DATA_FOUND 검사 (OPEN 은 NO_DATA 안 냄 — dead code, §6)
  ├─ for(ii=0;;ii++)  FETCH C1 INTO owKtp/ocode/ocodeNm
  │      NO_DATA_FOUND → break
  │      ii==max_row → i_isnext++, break (페이지 초과)
  │      zsedit → mod->orec[ii].{wKtp,code,codeNm}
  ├─ EXEC SQL CLOSE C1
  ├─ 결과 판정
  │      ii==0 && i_isnext==0 → e_msg "91001" (해당 자료 없음)
  │      i_isnext==1 → content_set_nkey, "53102"(내역 남음), eflg="0"
  │      else → "91003"(정상 조회), eflg="0"
  ├─ zredit(grid01_cnt, ii)
  └─ content->sndl += (ii-1)*sizeof(orec[0]); return RET_SUCCESS
```

## 4. 사용 DB 테이블

| 테이블 | 한글명 | 별칭 | 접근 | 용도 |
|---|---|---|---|---|
| `TB_FXB_CMC003D` | 공통코드상세 | `CMC003D1` | SELECT | `COM_CD_NM='FX_USER_DCD'` — FX사용자구분코드 목록 (wKtp='1') |
| `TB_FXB_CMC003D` | 공통코드상세 | `CMC003D2` | SELECT | `COM_CD_NM='SCRE_ACS_ATHR_DCD'` — 화면접근권한구분코드 목록 (wKtp='2') |

*DML 요약: 단일 커서 `W1701S05_C1` 이 동일 테이블 `TB_FXB_CMC003D` 를 `COM_CD_NM` 조건만 달리해 UNION ALL 로 2회 SELECT. INSERT/UPDATE/DELETE 없음. 공통함수(fnCustInfoQry 등) 경유 간접 접근 없음. `l_dbcommit`/`l_dbrollback` 은 순수 조회라 명목상.*

## 5. 로직 / 필터 규칙

- **입력 무관 고정 조회**: 입력 `fXuSerDcd` 를 검증/바인딩하지 않고 항상 두 코드그룹(`FX_USER_DCD`, `SCRE_ACS_ATHR_DCD`) 전량을 반환. 즉 권한그룹별 필터가 아님.
- **작업구분 분기 없음**: `switch(prGb)` 없이 단일 조회 경로. `switch` 는 진입부의 `ctyp→chnl` 매핑 뿐.
- **커서**: `W1701S05_C1` — UNION ALL 정적 쿼리, 바인드 변수 없음, ORDER BY 없음.
- **페이징**: `content->keyc`(K_NEXT/K_PREV) 로 `v_keyv` 를 만들지만 커서 WHERE 에 반영하지 않아 실제 이동 안 됨. `max_row` 초과 시 `i_isnext=1`, `content_set_nkey` 후 "53102" 안내만 함 → 다음 페이지 요청해도 동일 첫 페이지 재조회 (§6).
- **트랜잭션**: `W1701S05_F1` 이 `RET_SUCCESS` 면 `l_dbcommit`, 실패면 `l_dbrollback` (진입 함수에서 처리).
- **에러/메시지 코드 (e_msg.code → mesg)**:
  - `95000` 전산담당자 문의 (커서 DECLARE/OPEN 오류) — DECLARE 케이스는 return 주석처리라 실제 중단 안 함
  - `91001` 해당하는 자료가 없습니다 (0건)
  - `53102` 조회하실 내역이 남아 있습니다 (다음 페이지 존재, `eflg="0"`)
  - `91003` 정상적으로 조회되었습니다 (`eflg="0"`)
- `comhdr_o->rtyp` 는 진입부 "M" 설정 후 F1 에서 "G"(grid) 로 재설정.

## 6. 주의 — 문서/코드 불일치

| 위치 | 헤더/주석 표기 | 실제(.pc) |
|---|---|---|
| `.h` 헤더 Program Name | "메뉴상세조회" | `.pc` Description "코드명 조회". 공통코드(FX사용자구분/화면접근권한구분) 조회로 메뉴상세와 무관 |
| `W1701S05_F1` 함수 주석 | "나의 미래 예상현금흐름 입력-조회" | 실제는 공통코드명 조회. 타 서비스 복붙 오기 |
| Input `fXuSerDcd` (권한그룹코드) | 조회 조건 필드처럼 선언 | `.pc` 에서 전혀 참조 안 됨. 쿼리는 입력과 무관하게 두 그룹 전량 반환 (dead input) |
| Output `orec[].code[2]` | 코드 2B | 소스 커서 버퍼 `ocode[30+1]` 로 fetch 후 2B 필드로 zsedit → CMCD_VL 이 2B 초과 시 truncate 위험 |
| DECLARE 직후 SQLCODE 검사 (l.190) | 오류 시 처리 | DECLARE 는 비실행문이라 SQLCODE 무의미. `return` 도 주석처리 → 검사 자체가 무의미 (dead check) |
| OPEN 후 NO_DATA_FOUND 검사 (l.207) | 건수없음 처리 | OPEN 은 NO_DATA_FOUND 를 내지 않음 (FETCH 만) → 도달 불가 dead code |
| 페이징 (`v_keyv`/keyc) | K_NEXT/K_PREV 로 페이지 이동 의도 | `v_keyv` 를 커서 WHERE 에 바인딩하지 않아 페이지 이동 미동작. i_isnext=1 이어도 재요청 시 첫 페이지 재조회 |
| `content->sndl += (ii-1)*sizeof(orec[0])` (l.274) | — | `ii==0`(0건)일 때 orec 1개 크기만큼 음수 보정 → sndl 이 헤더 O_SZ 기준보다 orec 하나 작아짐. 무결성 검토 필요 |
| `max_row` 경계 | 마지막 유효 행 처리 | `ii==max_row` 에서 zsedit 전에 break → 그 행은 저장 안 하고 i_isnext 만 올림 (경계 1행 누락, 의도된 페이징 마커일 수 있음) |
