# 서비스 명세 — `W1411S03` (고객별 이용자조회(전체조회))

> 작성 양식: `docs/svc-spec-template.md`. 본 명세는 .pc 실 구현 기준이며, .h 헤더 주석과 어긋나는 부분은 §6 주의에 정리했다.

## 1. 개요

| 항목 | 값 |
|-----|-----|
| 서비스 코드 | `W1411S03` |
| 서비스 이름 | 고객별 이용자조회(전체조회) |
| 유형 | 조회 (SELECT only, 읽기 전용) |
| 구현 | `win/src/trn/W1400/W1411S03.pc` (298 라인) |
| I/O 헤더 | `win/src/inc/trn/W1411S03.h` |
| 라우팅 alias | `W1411S03` |
| 채널 | 매매 엔진 trn (broker `content->rkey` 기반 dispatch) |

특정 고객(전행고객실명대체번호)에 등록된 FX 이용자(사용자번호) 목록을 그리드로 전체 조회한다. DML 은 SELECT 커서 1개뿐이며, 데이터 변경은 없다.

## 2. I/O 전문 (`.h`)

**Input `W1411S03_I`**

| 필드 | 크기 | 설명 |
|-----|-----|-----|
| `prGb` | 1 | 작업구분 — `.pc` 에서 `'1'` 만 조회로 분기, 그 외 "작업구분 오류" |
| `wbcsRlnmAltrNo` | 16 | 전행고객실명대체번호 (조회 필수 키 — WHERE 등호 조건) |
| `fxUserNo` | 30 | FX사용자번호 (선택 — 값 있으면 등호, 없으면 NVL 로 전체) |

**Output `W1411S03_O`** (grid, 가변 orec)

| 필드 | 크기 | 설명 |
|-----|-----|-----|
| `grid01_cnt` | 6 | 그리드 건수 (반환 orec 수) |
| `orec[]` | (반복) | 아래 반복 구조 — `.h` 는 `orec[1]` 로 선언, 실제는 `MAX_RRB` 한도까지 가변 |
| ├ `orec[].wbcsRlnmAltrNo` | 16 | 전행고객실명대체번호 |
| └ `orec[].fxUserNo` | 30 | FX사용자번호 |

`sizeof(orec[0])` = 46 바이트. `max_row = MAX_RRB / 46`.

## 3. 구현 흐름 (`.pc`)

```
W1411S03()                                  // 진입 dispatch
  ├ prthdr / getmid / getmod               // COMHDR·입력·출력 주소 획득
  ├ e_msg 기본설정 (ltyp=KR, mtyp=S, rtyp=M)
  └ switch(mid->prGb[0])
       case '1' → W1411S03_F1()            // 조회
       default  → comhdr_o.mesg "작업구분 오류." + 즉시 return (commit/rollback 안 탐)
  ├ retc==RET_SUCCESS → eflg=0, l_dbcommit()
  │            else   → eflg=1, l_dbrollback()
  └ fnSetMsg(&e_msg, comhdr_o) → return content->sndl

W1411S03_F1()                               // 조회 본체
  ├ getmid/getmod, content->sndl += W1411S03_O_SZ, mod 0-clear
  ├ max_row = MAX_RRB / sizeof(orec[0])
  ├ switch(content->keyc)                   // 페이징 키 획득 (K_NEXT/K_PREV/최초)
  │    → vKeyv 채움 (※ 이후 SQL 에서 미사용 — §6)
  ├ 입력 rtrim: vWbcsRlnmAltrNo, vFxUserNo
  ├ DECLARE W1411S03_C1 CURSOR (SELECT … TB_FXB_CSC005R)
  ├ OPEN → SQLCODE 검사 (NO_DATA_FOUND→91001, 그외→10118 RET_FAILURE)
  ├ for(ii=0;;ii++) FETCH
  │    ├ NO_DATA_FOUND → break
  │    ├ 그외 error → CLOSE + 10118 + RET_FAILURE
  │    ├ ii==max_row → i_isnext++; break     // 한 건 더 있으면 next 표시
  │    └ orec[ii] 채움, vKeyv = cKeyv
  ├ CLOSE W1411S03_C1
  ├ ii==0 && i_isnext==0 → 91001 (자료 없음)
  │    else i_isnext==1  → content_set_nkey + 91002 (계속)
  │         else         → 91003 (완료) + grid01_cnt
  ├ zredit(grid01_cnt, ii)                   // 무조건 재설정
  └ content->sndl += (ii-1) * sizeof(orec[0]) → return RET_SUCCESS
```

## 4. 사용 DB 테이블

| 테이블 | 한글명 | 별칭 | 접근 | 용도 |
|-------|-------|-----|-----|-----|
| `TB_FXB_CSC005R` | 고객별 FX이용자 (전행실명대체번호+FX사용자번호) | `CSC005R` | SELECT | 커서 `W1411S03_C1` 로 고객별 이용자 목록 조회 |

*DML 요약: SELECT 1건 (커서 W1411S03_C1). INSERT/UPDATE/DELETE 없음. 공통함수 경유 간접 테이블 접근 없음. 트랜잭션은 읽기 전용이나 dispatch 성공 시 관례적으로 l_dbcommit() 호출.*

## 5. 로직 / 필터 규칙

- **작업구분**: `prGb='1'` 만 조회. 그 외는 "작업구분 오류." 메시지 후 커밋/롤백 없이 즉시 종료.
- **커서 필터**:
  - `WBCS_RLNM_ALTR_NO = RTRIM(:vWbcsRlnmAltrNo)` — 전행고객실명대체번호 등호(필수). 빈값이면 매칭 0건 → 91001.
  - `FX_USER_NO = NVL(:vFxUserNo, CSC005R.FX_USER_NO)` — 사용자번호가 오면 등호, 없으면 전체(자기 자신과 비교).
  - `ORDER BY CSC005R.FX_USER_NO`.
  - `KEY = WBCS_RLNM_ALTR_NO || FX_USER_NO` 를 조합해 페이징 키 후보로 FETCH.
- **페이징(설계상)**: `content->keyc` 가 `K_NEXT`/`K_PREV`/최초 에 따라 `vKeyv` 를 nkey/pkey/"0" 으로 세팅하고, FETCH 마다 `vKeyv=cKeyv` 로 갱신, `i_isnext==1` 이면 `content_set_nkey(vKeyv)` 로 다음 키를 내려보냄. **단, 커서 SQL 의 WHERE 절이 `vKeyv` 를 전혀 참조하지 않아 실제 페이징은 동작하지 않는다** (§6).
- **행 수 제한**: `ii==max_row` 시점(즉 max_row+1번째 FETCH 성공)에서 `i_isnext++` 후 break. orec 에는 `0..max_row-1` (max_row 건) 만 적재.
- **에러/메시지 코드** (`e_msg.code`, `fnSetMsg` 로 메시지 치환):
  - `91001` — 해당 자료 없음
  - `91002` — 조회 계속(다음 페이지 존재, NEXT KEY 내려감)
  - `91003` — 조회 완료
  - `10118` — 커서 OPEN/FETCH SQL 오류 → `RET_FAILURE` → `eflg=1` + rollback
- **eflg**: RET_SUCCESS → "0" + commit, 아니면 "1" + rollback.

## 6. 주의 — 문서/코드 불일치

| 위치 | 헤더/주석 표기 | 실제(.pc) |
|-----|--------------|-----------|
| `W1411S03_F1` 커서 WHERE | 상단 switch(keyc) 로 `vKeyv`(NEXT/PREV KEY) 를 구성하고 `content_set_nkey` 로 다음 키를 내려줌 → 페이징 지원처럼 보임 | 커서 SELECT 의 WHERE 절이 `vKeyv` 를 **전혀 사용하지 않음**. K_NEXT 요청도 항상 처음부터 동일 결과를 재조회 → 사실상 페이징 미동작 (dead 페이징 로직). |
| `W1411S03_F1` `content->keyc == K_PREV` | 이전 페이지 키 획득(`content_get_pkey`) 처리 존재 | pkey 를 세팅하는 곳이 없고(오직 nkey 만 set), vKeyv 자체도 WHERE 미사용이라 K_PREV 경로는 무의미. |
| `W1411S03()` OPEN 후 `SQLCODE==NO_DATA_FOUND` 분기 | 커서 OPEN 실패를 NO_DATA_FOUND 로 처리 (91001) | Oracle 커서 OPEN 은 통상 NO_DATA_FOUND 를 반환하지 않음(FETCH 에서 발생) → 사실상 dead 분기. 실제 무건수는 FETCH 후 `ii==0` 경로에서 91001 로 처리됨. |
| `content->sndl += (ii-1) * sizeof(orec[0])` | 반환 건수만큼 전문 길이 가산 | `ii==0`(무건수) 이면 `(ii-1)=-1` 로 orec 1건 크기만큼 **감산**. 앞서 `+= W1411S03_O_SZ`(orec[1] 포함) 했으므로 결과적으로 grid 헤더만 남아 정합은 맞으나, 음수 곱 의존은 취약(가독성/방어 부재). |
| `grid01_cnt` 세팅 | 완료(91003) 분기에서 1회 | 91003 분기(286행)와 무조건 라인(293행)에서 **중복 zredit** — 무해하나 잉여. |
| `W1411S03()` 지역변수 `mod` | 출력 자료구조 사용 목적 | main 함수 내 `mod` 는 getmod 로 받되 실제 사용 안 함(dead). 실제 orec 적재는 `W1411S03_F1` 에서 별도 getmod. |
| 주석 "CSM004M 상태코드 볼필요없다 … CSC005R" | 비활성 상태면 CSC005R 에 데이터 없음 전제 | 코드상 상태 테이블 조인/조회 없음 — 주석은 전제 설명일 뿐 실제 검증 로직 부재. |
