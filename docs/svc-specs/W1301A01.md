## 3. 구현 흐름 (`.pc`)

```
W1301A01()                      ← 진입 (콜백)  ※ 함수 주석은 "조회"지만 실제 등록/수정 dispatch
  ├─ prthdr / getmid / getmod   COMHDR + input(mid) + output(mod) 준비
  ├─ E_MSG 기본설정              ltyp="KR", mtyp="S"(FXON), rtyp="M"(Message전송)
  ├─ switch(mid->prGb)
  │     '2' | '3' → retc = W1301A01_F1(...)    ← 등록/수정 실제 로직
  │     그외      → comhdr_o->mesg="작업구분 오류." 후 즉시 return (MSG_NOWORK)
  └─ retc==RET_SUCCESS ? l_dbcommit()+eflg"0" : l_dbrollback()+eflg"1"
        (명시적 커밋, AUTOCOMMIT 아님)  →  fnSetMsg(&e_msg, comhdr_o) → return sndl

W1301A01_F1()                   ← 등록/수정 (함수 주석은 "등록"이지만 둘 다 처리)
  ├─ getmid/getmod, memset(mod)
  ├─ 입력 바인딩: yprintf + l_rtrim → vCrcd, vCrcdNm, vCrcdRefVl, vDcpnPcsnDcd, vUseYn
  │              l_n2i → vDcpuRcn, vRpsnSqc ; vFxSysLsmdId ← comhdr_i->usid
  ├─ 유효성 (순차, 실패 시 RET_FAILURE)
  │     vCrcd        공백 → e_msg "10190" (통화 확인)
  │     vCrcdNm      공백 → e_msg "10191" (통화명 확인)
  │     vDcpnPcsnDcd 공백 → e_msg "10188" (Rounding Type 확인)
  │     vUseYn       공백 → e_msg "32517" (사용여부 확인)
  ├─ prGb=='3' → EXEC SQL UPDATE TB_FXB_CMG005M ... WHERE CRCD=:vCrcd
  │     SQLCODE==NO_DATA_FOUND → e_msg "10221" (수정할 데이터 없음)   ※ §6: UPDATE 에선 미발생
  │     그 외 SQLCODE!=0       → e_msg "10122" (수정 중 오류) → RET_FAILURE
  ├─ prGb=='2' → EXEC SQL INSERT INTO TB_FXB_CMG005M ...
  │     SQLCODE==DUP_KEY → e_msg "10223" (통화 중복)
  │     그 외 SQLCODE!=0  → e_msg "10122" (등록 중 오류) → RET_FAILURE
  └─ 정상 → e_msg "10150" (MSG_REQUERY) → RET_SUCCESS
```

## 4. 사용 DB 테이블

| 테이블 | 한글명 | 별칭 | 접근 | 용도 |
|---|---|---|---|---|
| `TB_FXB_CMG005M` | 통화기본 (통화 마스터) | CMG005M | **UPDATE** (prGb='3') / **INSERT** (prGb='2') | 통화코드(`CRCD`) 기준 통화명/참조값/소수점처리/자릿수/표시순서/사용여부 등록·수정. 변경자(`FX_SYS_LSMD_ID`)·변경시각(`FX_SYS_LSMD_TS=SYSDATE`) 스탬프 |

*DML 요약: `TB_FXB_CMG005M` 에 처리구분별로 UPDATE 1문 또는 INSERT 1문. SELECT/DELETE/커서/조인 없음. 공통함수 경유 간접 테이블 접근 없음.*

- **UPDATE 컬럼** (`WHERE CRCD=RTRIM(:vCrcd)`): `CRCD_NM=RTRIM(:vCrcdNm)`, `CRCD_REF_VL=NVL(:vCrcdRefVl,'O')`, `DCPN_PCSN_DCD`, `DCPU_RCN=NVL(:,0)`, `RPSN_SQC=NVL(:,0)`, `USE_YN=NVL(:,'N')`, `FX_SYS_LSMD_ID`, `FX_SYS_LSMD_TS=SYSDATE`
- **INSERT 컬럼**: `CRCD`, `CRCD_NM`, `CRCD_REF_VL=NVL(:,'O')`, `DCPN_PCSN_DCD`, `DCPU_RCN`, `RPSN_SQC=NVL(:,0)`, `USE_YN=NVL(:,'Y')`, `FX_SYS_LSMD_ID`, `FX_SYS_LSMD_TS=SYSDATE`

## 5. 로직 / 필터 규칙

- **작업구분 분기**: `switch(mid->prGb[0])` — `'2'`(등록)/`'3'`(수정) 만 `W1301A01_F1` 진입. 나머지는 처리 없이 "작업구분 오류." 로 return.
- **입력 유효성**: `crcd`/`crcdNm`/`dcpnPcsnDcd`/`useYn` 4개는 공백/NULL 이면 각각 `10190`/`10191`/`10188`/`32517` 로 거부. `crcdRefVl`/`dcpuRcn`/`rpsnSqc` 는 미검증(선택, SQL 에서 NVL 기본값 적용).
- **DML 분기**: 동일 함수에서 `prGb=='3'` UPDATE / `prGb=='2'` INSERT. 두 블록이 독립 if 이므로 어느 쪽도 아니면 DML 없이 성공 처리로 통과 (단, dispatch 단계에서 이미 '2'/'3' 만 진입).
- **에러 매핑** (`e_msg.code`→`mesg`):
  - `10221` 수정할 데이터 없음(NO_DATA_FOUND) — §6 참고(UPDATE 경로에서 실질 미발생)
  - `10223` 등록 중복(DUP_KEY), `10122` 등록/수정 중 오류, `10150` 정상 완료
- **트랜잭션**: `W1301A01_F1` 은 DML 만 수행, 커밋/롤백은 상위 `W1301A01` 이 `retc` 로 결정. `RET_SUCCESS→l_dbcommit`+`eflg"0"`, 실패→`l_dbrollback`+`eflg"1"`. AUTOCOMMIT 아님 — 명시적 커밋 필수.
- **결과 전달**: 데이터 없이 COMHDR — `eflg`(0 정상/1 오류) + `e_msg.code`→`mesg`.
  - → WTG `/v1/tx` 는 `errn`≠0 이어도 전문 있으면 200, `errn`/COMHDR 본문 동봉(레거시 규약).

## 6. 주의 — 문서/코드 불일치 (마이그레이션 시 확인)

| 위치 | 헤더/주석 표기 | 실제(`.pc`) |
|---|---|---|
| `W1301A01()` 함수 주석 | "조회" | 등록/수정 dispatch (조회 로직 없음) |
| `W1301A01_F1()` 함수 주석 | "등록" | 등록(INSERT)+수정(UPDATE) 둘 다 처리 |
| `.pc` L72 인라인 주석 | "처리구분 (1:조회, 2:등록, 3수정, 4삭제)" | 실제 switch 는 `'2'`/`'3'` 만 처리, 조회/삭제 미구현 |
| UPDATE 후 `SQLCODE==NO_DATA_FOUND` 분기 (L232, e_msg 10221) | 수정할 데이터 없음 처리 | Oracle UPDATE 는 매칭 0건이어도 `SQLCODE=0`(NOT_FOUND 아님) — **사실상 dead branch**. 존재하지 않는 CRCD 수정 시 오류 없이 커밋되어 "정상" 응답. 실제 갱신 건수 검증하려면 `SQLERRD[2]`(rows processed) 확인 필요 |
| UPDATE `USE_YN=NVL(:vUseYn,'N')` vs INSERT `USE_YN=NVL(:vUseYn,'Y')` | — | 미입력 시 기본값 불일치(수정 'N' / 등록 'Y'). 단 `useYn` 은 상위에서 필수 검증(공백 시 32517)되므로 NVL 은 실질 미발동 |
| `.h` `dcpuRcn` 크기 5 / `rpsnSqc` 크기 7 | 문자열 폭 | `.pc` 에서 `l_n2i` 로 int 변환 후 바인딩 (`vDcpuRcn`,`vRpsnSqc`) |

> WTG 는 `.h` 를 파싱해 필드명/크기만 취하고 전문을 통과시키므로 위 주석 오기가
> 동작에 영향 없다. 단 `10221`(수정 데이터 없음) dead branch 는 존재하지 않는 통화코드
> 수정 요청이 "성공"으로 응답되는 UX 이슈이므로 마이그레이션/신규 구현 시 rows-affected
> 검증 추가를 검토할 것.
