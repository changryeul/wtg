|---|---|---|---|---|
| `TB_FXB_CSC001M` | 고객약정정보기본 | — | **SELECT** | `WBCS_RLNM_ALTR_NO` 로 `OPN_BRCD`(사무실/계좌관리사무소코드) 단건 조회 → API 입력 |

## 5. 로직 / 필터 규칙

- **DML**: 임시테이블(`TB_FXB_CMG062M`)을 고객키 단위로 DELETE→INSERT 후, 등록원장(`TB_FXB_CSC011M`)과 FULL OUTER JOIN 커서로 최종 grid 산출. 성공 시 `l_dbcommit`(temp DML 확정), 실패 시 `l_dbrollback`.
- **조회 체인**: ① `TB_FXB_CSC001M` 에서 사무소코드 확보 → ② CORE BANKING `FEFNPL95R0`(inq_dsc="5", tr_dsc="2" 조회) 로 SUMMIT 측 펀드목록 수신 → ③ 목록을 임시테이블에 적재 → ④ 임시테이블 ⨝ 등록원장으로 등록상태 비교.
- **등록상태(comment) 산정** (커서 CASE):
  - `기등록`: 등록원장·임시(SUMMIT) 양쪽 존재
  - `SUMMIT없음`: 등록원장에만 존재
  - `미등록`: 임시(SUMMIT)에만 존재
  - `알수없음`: 양쪽 NULL (FULL OUTER JOIN 특성상 사실상 미발생)
- **입력 유효성**: `wbcsRlnmAltrNo` 공백 시 `32017`+`eflg=1` 로 실제 거부 (W1119 와 달리 실검증 존재).
- **GRID/페이징**: `content->keyc`(NEXT/PREV/최초)로 `nextKey` 를 만들어 API 요청 `itnbnk_cus_id` 에 넘긴다. 그러나 최종 커서는 임시테이블 전체를 `wbcsRlnmAltrNo` 로만 재조회하며 `nextKey` 미사용, API 응답 `nx_data_yn` 도 출력에 반영 안 함 → 클라이언트로 next-key 미전달, 실질 페이징 미동작(§6).
- **트랜잭션**: `RET_SUCCESS→l_dbcommit / 실패→l_dbrollback`. AUTOCOMMIT 아님.
- **에러/결과 코드**:
  - 성공: `91003` "조회 완료", `eflg=0`.
  - 입력 공백: `32017` "입력값[고객번호] 확인", `eflg=1`.
  - 고객(사무소) 없음: `82030`, `eflg=1`.
  - 조회 SQL 오류: `10118`.
  - 임시테이블 DELETE/INSERT 오류: `10122`.
  - API 오류: `43060`(호출 실패/포맷) / `88888`(뱅킹 업무오류 메시지) — `fefnpl95r0` 내부 set, 상위서 `eflg=1`.
  - 결과 0건: `91001` set 후 즉시 `91003` 으로 덮임 → 실제로는 항상 "조회 완료" 로 응답(§6).
- **결과 전달**: grid(N×4필드) + COMHDR(`eflg`/`e_msg.code`→`mesg`). WTG `/v1/tx` 는 `errn`≠0 이어도 전문 있으면 200, COMHDR 본문 동봉(레거시 규약).
