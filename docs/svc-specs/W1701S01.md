# 서비스 명세 — `W1701S01` (메뉴정보조회)

> 작성 양식: `docs/svc-spec-template.md`. 본 명세는 `.pc` 실 구현 기준이며, `.h` 헤더 주석과 어긋나는 부분은 §6 주의에 정리했다.

## 1. 개요
| 항목 | 값 |
|-----|-----|
| 서비스 코드 | `W1701S01` |
| 서비스 이름 | 메뉴정보조회 (메뉴 계층 트리 조회) |
| 유형 | 조회 (SELECT only, 그리드 페이징) |
| 구현 | `win/src/trn/W1700/W1701S01.pc` (357 줄) |
| I/O 헤더 | `win/src/inc/trn/W1701S01.h` |
| 라우팅 alias | `W1701S01` |
| 채널 | Web/HTS/MTS/EMP/API 공용 (comhdr `ctyp` 로 `chnl` 세팅, `rtyp="G"` 해외선물 메시지코드) |

메뉴 마스터(`TB_FXB_CMC010M`)를 상위메뉴ID 기준으로 재귀 전개(WITH ... UNION ALL)해 특정 레벨의 메뉴 목록을 그리드로 반환한다. 사용자구분(`TB_FXB_CMC011M`)은 outer join 으로 필터링.

## 2. I/O 전문 (`.h`)

**Input `W1701S01_I`** (9 byte)
| 필드 | 크기 | 설명 (실제 의미) |
|-----|-----|-----|
| `mEnuLvlVl` | 1 | 메뉴레벨 — 최종 SELECT 의 `MENU_LVL_VL = :imEnuLvlVl` 필터 |
| `hGrnMenuId` | 6 | 상위메뉴ID — 재귀 CTE 시작점 `A.HGRN_MENU_ID = :ihGrnMenuId` |
| `fXuSerDcd` | 2 | 사용자구분코드 — `B.FX_USER_DCD(+) like '%'||:ifXuSerDcd` |

**Output `W1701S01_O`** (grid — `orec` 가변, 실제 최대 30건)
| 필드 | 크기 | 설명 |
|-----|-----|-----|
| `grid01_cnt` | 6 | 그리드01 건수 (반환 행 수 `ii`) |
| `orec[].mEnuId` | 6 | 메뉴ID |
| `orec[].mEnuNm` | 100 | 메뉴명 |
| `orec[].hGrnMenuId` | 6 | 상위메뉴ID |
| `orec[].mEnuRpsnSqn` | 5 | 화면표시순서(=메뉴정렬순서 `MENU_RPSN_SQN`) |
| `orec[].mEnuLvlVl` | 1 | 메뉴레벨 |
| `orec[].uSeYn` | 1 | 사용여부 |
| `orec[].mEnuDescCon` | 100 | 메뉴정보내용 |

`.h` 는 `orec[1]` 로 선언되어 있고, `.pc` 가 `content->sndl` 을 실 건수만큼 늘려 가변 전송한다(§6 sndl 계산 주의).

## 3. 구현 흐름 (`.pc`)
```
W1701S01()                         # 엔트리
  getmid / getmod                  # 입출력 주소 획득
  e_msg 초기화 (pname=rkey, ltyp, mtyp="S", rtyp="M")
  switch(comhdr_i->ctyp[0])        # HTS/MTS/WTS/EMP/API/SPI → comhdr_i->chnl 세팅
  retc = W1701S01_F1(...)          # 실 조회
  retc==RET_SUCCESS ? l_dbcommit() : l_dbrollback()
  fnSetMsg(&e_msg, comhdr_o)       # 응답 코드 세팅
  return content->sndl

W1701S01_F1()                      # 조회 본체
  getmid/getmod, sndl += O_SZ, memset(mod)
  max_row = 30 (하드코딩 "임시설정")
  rtyp="G"
  switch(content->keyc)            # 페이징 키
    K_NEXT → content_get_nkey → v_keyv
    K_PREV → content_get_pkey → v_keyv
    default → v_keyv="0"
  입력값 yprintf/l_rtrim (imEnuLvlVl, ihGrnMenuId, ifXuSerDcd)
  DECLARE W1701S01_C1  (재귀 CTE, §4/§5)
    SQLCODE!=0 → e_msg.code="95000" (return 은 주석처리 — 계속 진행)
  OPEN  W1701S01_C1
    SQLCODE!=0 → e_msg.code="95000", return RET_FAILURE
    (SQLCODE==NO_DATA_FOUND → return RET_FAILURE  ※ dead code, §6)
  for(ii=0; ; ii++)
    FETCH INTO 로컬버퍼들
    NO_DATA_FOUND → break
    if (ii==max_row) { i_isnext++; break; }   # 31번째 fetch 시 저장 없이 종료
    zsedit → mod->orec[ii].*                    # 30건까지 저장
    v_keyv = ohGrnMenuId+omEnuRpsnSqn+omEnuId    # NEXT KEY (§6 버그)
  CLOSE W1701S01_C1
  if (ii==0 && i_isnext==0) e_msg.code="91001"   # 자료 없음
  else if (i_isnext==1) content_set_nkey, code="53102"(더 있음), eflg="0"
       else            code="91003"(정상조회), eflg="0"
  zredit(grid01_cnt, ii)
  content->sndl += (ii-1)*sizeof(orec[0])
  return RET_SUCCESS
```
트랜잭션: 조회 전용이나 관례상 성공 시 `l_dbcommit()`, 실패 시 `l_dbrollback()`.

## 4. 사용 DB 테이블
| 테이블 | 한글명(추정) | 별칭 | 접근 | 용도 |
|-----|-----|-----|-----|-----|
| `TB_FXB_CMC010M` | 메뉴 마스터 | A / NPLUS1 | SELECT | 메뉴 계층(상위-하위) 재귀 전개 원천 |
| `TB_FXB_CMC011M` | 메뉴 사용자구분(권한) 매핑 | B | SELECT (outer join) | `FX_USER_DCD` 로 사용자구분코드 필터 |

*DML 요약: SELECT 전용. INSERT/UPDATE/DELETE 없음. 커서 `W1701S01_C1` 1개(recursive CTE) OPEN→FETCH 루프→CLOSE.*

## 5. 로직 / 필터 규칙
- **재귀 CTE**: `WITH LAYER_1 AS ( <anchor: HGRN_MENU_ID=:ihGrnMenuId, CMC010M A ⟕ CMC011M B on MENU_ID & FX_USER_DCD like '%'||:ifXuSerDcd> UNION ALL <recursive: NPLUS1.HGRN_MENU_ID = LAYER_1.MENU_ID> )` → 최종 `SELECT ... FROM LAYER_1 WHERE MENU_LVL_VL=:imEnuLvlVl AND HGRN_MENU_ID||LPAD(MENU_RPSN_SQN,5,'0')||MENU_ID > :v_keyv ORDER BY MENU_ID, HGRN_MENU_ID, TO_NUMBER(MENU_RPSN_SQN), MENU_ID`.
- **페이징**: keyc(K_NEXT/K_PREV/최초)로 `v_keyv` 결정, 최초는 "0". `HGRN_MENU_ID||LPAD(RPSN,5)||MENU_ID > v_keyv` 로 다음 페이지 절단. 페이지 크기 `max_row=30` 고정.
- **다음 존재 표시**: 31번째 행 fetch 성공 시 `i_isnext=1`, `content_set_nkey`, `e_msg.code="53102"`.
- **에러/메시지 코드**:
  - `95000` — 쿼리 오류(DECLARE/OPEN 실패).
  - `91001` — 해당 자료 없음 (ii==0 && i_isnext==0).
  - `53102` — 조회하실 내역이 남아 있습니다 (다음 페이지 존재), `eflg="0"`.
  - `91003` — 정상적으로 조회되었습니다, `eflg="0"`.

## 6. 주의 — 문서/코드 불일치
| 위치 | 헤더/주석 표기 | 실제(.pc) |
|-----|-----|-----|
| `.pc` L112-113 `W1701S01_F1` 주석 | "나의 미래 예상현금흐름 입력-조회" | 실제는 메뉴정보(계층) 조회 — 타 서비스에서 복붙한 오기 |
| `.h` `orec.mEnuRpsnSqn` 주석 | "화면표시순서" | SQL/`.pc` 로컬은 `MENU_RPSN_SQN` = "메뉴정렬순서" (동일 값, 명칭만 상이) |
| `.pc` L327 NEXT KEY 생성 | — | **버퍼 오버플로 버그**: `strcat(strcat(ohGrnMenuId, omEnuRpsnSqn), omEnuId)` 가 7바이트(`char[6+1]`) `ohGrnMenuId` 버퍼에 최대 17바이트를 이어붙임 → 스택 손상 위험 |
| `.pc` L247 vs L327 | — | **페이징 키 포맷 불일치**: SQL 비교키는 `LPAD(MENU_RPSN_SQN,5,'0')` 로 5자리 zero-pad 하나, NEXT KEY 는 `omEnuRpsnSqn`(TO_CHAR 원본, 미패딩)로 조립 → K_NEXT 절단 비교가 어긋나 페이징 오작동 가능 |
| `.pc` L267-271 OPEN 후 `if(SQLCODE==NO_DATA_FOUND)` | — | **dead code**: 커서 OPEN 은 NO_DATA_FOUND 를 반환하지 않음. 실제 무자료 판정은 fetch 루프 후 `ii==0` 로만 이뤄짐 |
| `.pc` L250-255 DECLARE 실패 처리 | e_msg 95000 세팅 | `return(RET_FAILURE)` 가 주석처리 되어 실패로도 OPEN 을 강행 |
| `.pc` L157 `max_row` | (주석 "임시설정") | 원래 `MAX_RRB/sizeof(orec)` 계산이 주석처리되고 30 하드코딩 — 전문 버퍼 상한과 무관하게 30건 고정 |
| `.pc` L354 `content->sndl += (ii-1)*sizeof(orec[0])` | — | `O_SZ`(orec 1개 포함)에 `(ii-1)` 더해 총 ii건 길이를 맞추는 계산. ii==0(무자료) 시 `-1` 로 orec 1개분이 빠져 헤더+grid_cnt 크기만 전송(동작상 무해하나 의도 불명확) |
| `.pc` L248 ORDER BY | — | `MENU_ID` 가 두 번 등장(중복 정렬키, 무해) |
| `content->sndl` / K_PREV | — | K_PREV 분기는 pkey 를 읽지만 CTE 는 단방향 `> :v_keyv` 만 사용 → 이전 페이지 역방향 처리 로직 부재(사실상 next 전용 페이징) |
