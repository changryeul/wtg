1. 목적 · 범위

forwarder + mci-price를 실시간 FX 시세 배포 엔진으로 자리매김한다 — 수집 → best 산정 → 마진 → 배포까지. MDS(C 시장데이터 시스템)의 완전 대체가 아니라, 라이브 시세 가공·배포 역할만 WTG로 가져오는 것이 목표다.

전체 tick의 DB 영속(SOR)이나 query-server 조회 카탈로그 같은 무거운 책임은 범위에서 제외한다. 핵심 관심사는 영속이 아니라 안정성·정합성·저지연이다.

핵심 관점

이 범위에서 forwarder + mci-price는 이미 거의 완성형이다. 앞으로 할 일은 기능 추가가 아니라 "배포 엔진으로서 믿을 만한가"를 굳히는 신뢰성 작업(3장)이다.

3. 신뢰성 요건

배포 엔진의 품질은 "유실 없이, 막히지 않고, 정확하게, 보이게" 흐르느냐로 결정된다. 6개 요건과 현 상태를 정리한다.

3.1 broker를 hot path에서 제거 대부분 완료

broker publisher는 단일 thread라 고부하에서 천장(SIGABRT)에 걸린다(부하테스트 64k tick/s → delivery ~62%). 시세 수집 경로에서 가장 큰 불안정 요인이다.

현 상태

forwarder --publish-mode grpc로 broker를 우회해 mci-price에 직접 push 가능. mci-price는 broker subscribe와 동일한 IngestEnvelopes hot path로 진입. 모드만 grpc로 확인/고정하면 이 요건은 충족.

3.2 conflation 기반 backpressure 보강

시세는 매매와 달리 최신값만 의미가 있다. 소비자가 뒤처지면 중간 tick을 버리고 최신만 유지해야 하며, 절대 생산자를 block하지 않아야 한다.

3.3 정합성 — Source 누락 방지 점검

조용한 유실 지점

BestConsumer는 Tick.Source == ""이면 그 tick을 버린다. forwarder가 envelope의 src를 안 채우면 received·conflation은 올라가지만 best·봉·마진·배포 어디에도 안 들어간다. 이 drop은 price-stats.dropped에 안 잡힌다.

판정자는 GET /v1/best-stats — 여기에 feed의 source(예: SMB)가 (symbol, source)로 보이면 정상 ingest. 안 보이면 Source 누락 drop이다. 요건: forwarder가 모든 envelope에 src를 채우는 것을 계약으로 강제 + best-stats source 모니터.

3.4 유실 가시화 — seq gap 신규

UDP는 재전송이 없어 손실은 막을 수 없다. 대신 envelope의 seq로 누락 구간을 메트릭으로 노출하면 "조용한 유실"이 "보이는 유실"이 된다. per-feed seq gap 카운터를 추가해 어느 feed가 얼마나 빠졌는지 관측.

3.5 HA · 샤딩 필요 시

mci-price/edge 다중 인스턴스 + 자동 재연결(이미 있음). 단일 인스턴스가 포화하면 심볼 단위 샤딩으로 처리 여유 확보(etcd 일관성, docs/mci-price-ha.md). 배포 엔진은 best 계산이 O(profile)이라 보통 한 인스턴스로도 여유가 크다 — 샤딩은 실제 포화 시에만.

4. 보관 정책 (옵션 — 미정)

배포 엔진은 raw tick을 영속할 이유가 적다. 3-tier로 나눠 필요한 만큼만 둔다.

tier저장용도 / 비고
최신값conflation (메모리)현재가 조회·배포. 항상 필요, 이미 있음
봉(OHLC)TimescaleDB (영속)차트·조회. 현 Archiver 그대로(1m+)
raw tick보관 안 함 / 짧은 ring기본은 미보관. 디버그·재전송 확인용 최근 N초 ring만 선택적
권고

최신값 + 봉이면 "배포 엔진 + 차트/조회"는 충분하다. raw tick 영속은 두지 않는 것을 기본값으로. 추후 규제/정산 요구가 생기면 그때 MDS의 SOR를 쓰거나 선택적 영속을 더한다 — 보관 결정이 배포 엔진 설계를 막지 않는다.

그림 6-1. 안정성 우선 → 확장 → 보관 순
  1. 1단계 (즉시) — forwarder를 gRPC 직결 모드로 고정하고, /v1/best-stats로 feed source가 실제 ingest되는지 확인(Source 누락이면 forwarder src 수정). 이 둘만으로 "받지만 안 들어가는" 현 우려가 해소된다.
  2. 2단계 (단기) — PricingConsumer 비동기 buffer ON + 버퍼 튜닝, 어떤 경로도 block 없도록 점검, per-feed seq gap 카운터 추가.
  3. 3단계 (중기) — 부하에 따라 mci-price/edge 다중 인스턴스 + 심볼 샤딩, seq gap·source 대시보드.
  4. 4단계 — 보관 정책 확정(최신+봉 기본, raw 미보관/ring). 규제 요구 발생 시 SOR 연동 검토.
