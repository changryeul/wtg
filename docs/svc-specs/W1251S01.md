# 서비스 명세 — `W1251S01` (코드정보기본 조회)

> 작성 양식: `docs/svc-spec-template.md`. 본 명세는 `.pc` 실 구현 기준이며, `.h` 헤더 주석과 어긋나는 부분은 §6 주의에 정리했다.

## 1. 개요

| 항목 | 값 |
|-----|-----|
| 서비스 코드 | `W1251S01` |
| 서비스 이름 | 코드정보기본 조회 (공통코드 기본정보 목록 조회) |
| 유형 | 조회 (SELECT 전용, 그리드 페이징) |
| 구현 | `win/src/trn/W1250/W1251S01.pc` (336 라인) |
| I/O 헤더 | `win/src/inc/trn/W1251S01.h` |
| 라우팅 alias | `W1251S01` |
| 채널 | `ChannelWeb` (`/v1/tx` 경유) |
| 작성 | 2023-12 오호림 |

`prGb` 작업구분으로 분기하나 현재 구현은 `'1'`(조회) 단일 케이스만 존재. 코드정보기본 테이블(`TB_FXB_CMC002M`)을 공통코드명/한글명/사용여부 필터로 조회해 그리드로 반환한다.

## 3. 구현 흐름 (`.pc`)

```
W1251S01(진입)                                    # 라인 42
  ├─ prthdr / getmid / getmod                     # comhdr·mid·mod 주소 획득
  ├─ e_msg 기본설정 (ltyp=KR, mtyp=S, rtyp=M)
  ├─ switch(mid->prGb[0])                          # 작업구분 분기
  │    case '1' → retc = W1251S01_F1(...)          # 조회
  │    default  → comhdr_o.mesg="작업구분 오류." 즉시 return
  ├─ retc==RET_SUCCESS ? l_dbcommit(), eflg="0"
  │                    : l_dbrollback(), eflg="1"  # 조회 전용이나 규약상 commit/rollback 수행
  ├─ fnSetMsg(&e_msg, comhdr_o)                    # e_msg.code → mesg/rcod/pflg 세팅
  └─ return content->sndl

W1251S01_F1(조회)                                  # 라인 118
  ├─ getmid/getmod, sndl += W1251S01_O_SZ, mod 전체 memset
  ├─ max_row = MAX_RRB / sizeof(orec[0])           # 한 화면 최대 건수
  ├─ switch(content->keyc)                          # 페이징 키
  │    K_NEXT → content_get_nkey → vKeyv
  │    K_PREV → content_get_pkey → vKeyv
  │    default(최초) → vKeyv="0"
  ├─ yprintf/l_rtrim 로 vComCdNm/vCmcdKrnNm/vUseYn 준비
  ├─ EXEC SQL DECLARE W1251S01_C1 CURSOR (TB_FXB_CMC002M)
  ├─ EXEC SQL OPEN W1251S01_C1
  │    OPEN error → e_msg.code=10118 (단, return 안 함 — §6)
  │    NO_DATA_FOUND → e_msg.code=91001 (OPEN 시 사실상 dead — §6)
  ├─ for(ii=0;;ii++)                                # FETCH 루프
  │    FETCH INTO c-변수들
  │    SQLCODE==NO_DATA_FOUND → break
  │    SQLCODE 기타오류 → CLOSE, e_msg.code=10118, return RET_FAILURE
  │    ii==max_row → i_isnext++, break              # 한 건 초과 fetch 로 다음 유무 판정
  │    zsedit 로 mod->orec[ii] 채움, vKeyv=cKeyv    # NEXT KEY = COM_CD_NM
  ├─ EXEC SQL CLOSE W1251S01_C1
  ├─ ii==0 && i_isnext==0 → e_msg.code=91001 (자료없음)
  │  else i_isnext==1 → content_set_nkey, e_msg.code=91002 (계속)
  │       else        → e_msg.code=91003 (완료)
  ├─ zredit(mod->grid01_cnt, ii)
  └─ content->sndl += (ii-1) * sizeof(orec[0]); return RET_SUCCESS
```

## 4. 사용 DB 테이블

| 테이블 | 한글명 | 별칭 | 접근 | 용도 |
|-------|-------|-----|-----|-----|
| `TB_FXB_CMC002M` | 코드정보기본(공통코드기본) | `CMC002M` | SELECT | 커서 `W1251S01_C1` 로 공통코드 목록 조회 |

*DML 요약: SELECT 1건 (커서 조회) 뿐. INSERT/UPDATE/DELETE 없음. 조회 전용이나 진입 함수가 규약상 `l_dbcommit`/`l_dbrollback` 을 호출한다. 공통함수 경유 간접 테이블 접근 없음.*

## 5. 로직 / 필터 규칙

- **작업구분(prGb)**: `'1'` 만 유효. 그 외는 `comhdr_o.mesg="작업구분 오류."` + `MSG_NOWORK` 로그 후 즉시 반환.
- **페이징**: `content->keyc` 로 최초/NEXT/PREV 구분. NEXT_KEY 는 `COM_CD_NM`. 커서 조건 `COM_CD_NM > :vKeyv` + `ORDER BY COM_CD_NM` (오름차순). 최초는 `vKeyv="0"`.
- **다음 페이지 판정**: `max_row = MAX_RRB / sizeof(orec[0])`. 루프에서 `ii==max_row` 도달 시 한 건 더 fetch 된 상태로 `i_isnext++` 후 break — 저장은 `max_row` 건만.
- **동적 필터 (WHERE)**:
  - `vComCdNm` 이 비었으면 → `CMCD_KRN_NM = NVL(:vCmcdKrnNm, CMCD_KRN_NM)` (한글명 정확일치, 빈값이면 전체).
  - `vComCdNm` 이 있으면 → `CMCD_KRN_NM LIKE '%'||vCmcdKrnNm||'%'` **OR** `COM_CD_NM LIKE '%'||UPPER(vComCdNm)||'%'`.
  - `USE_YN = NVL(:vUseYn, USE_YN)` — 사용여부 필터(빈값이면 전체).
- **트랜잭션**: `W1251S01_F1` 성공 시 진입 함수가 `l_dbcommit`+`eflg="0"`, 실패 시 `l_dbrollback`+`eflg="1"`.
- **에러/메시지 코드 (e_msg.code → fnSetMsg)**:
  - `10118` — 자료 조회중 오류 (커서 OPEN/FETCH 오류).
  - `91001` — 해당하는 자료가 없습니다 (ii==0 && i_isnext==0).
  - `91002` — 조회가 계속됩니다 (i_isnext==1, NEXT_KEY 세팅).
  - `91003` — 조회가 완료되었습니다 (마지막 페이지).
- **전송 길이**: `sndl += W1251S01_O_SZ` (헤더+orec 1건 포함) 후 `sndl += (ii-1)*sizeof(orec[0])` 로 실제 건수만큼 보정.

## 6. 주의 — 문서/코드 불일치

| 위치 | 헤더/주석 표기 | 실제(`.pc`) |
|-----|-------------|-----------|
| `.pc:48` | `mod` 주석 "입력값 주소" | 실제는 **출력값** 주소 (`getmod`). 복붙 오기 |
| `.pc:186` | `memset(vComCdNm, 0x00, sizeof(vCmcdKrnNm))` | 두 번째 memset 이 대상 변수를 `vComCdNm` 으로 잘못 지정 — `vCmcdKrnNm` 은 memset 되지 않음. 뒤이어 `yprintf` 가 덮어써 실질 영향은 없으나 명백한 복붙 버그 |
| `.pc:229-233` | OPEN 오류 시 `e_msg.code=10118` 세팅 | **return 하지 않고** FETCH 루프로 진행. 이후 정상 코드(91001/91002/91003)로 덮어써져 10118 이 소실될 수 있음 |
| `.pc:235-239` | OPEN 직후 `NO_DATA_FOUND` → `91001` | 커서 OPEN 은 통상 `NO_DATA_FOUND` 를 반환하지 않음 — 사실상 dead code (자료없음 판정은 라인 307 에서 별도 수행) |
| `.pc:172-174` (K_PREV) | 이전 페이지 조회 | 커서 조건이 `COM_CD_NM > :vKeyv` + ORDER BY ASC 로 항상 **전방(forward) 페이징만** 동작. PREV 키로 진입해도 역방향 조회가 되지 않음 (사실상 이전 페이지 미동작) |
| `.pc:183` 주석 | "입력항목 유효성 check" | 실제 유효성 검증 로직 없음 — trim/로그만 수행 (dead comment) |
| `.pc:218` | 필터 의미 | `vComCdNm` 유무 분기가 코드명/한글명 LIKE 를 혼합 적용. 헤더의 `comCdNm`(공통코드명) 단독 필터 기대와 달리 한글명 조건과 OR 결합됨. 주석 처리된 라인 220-221 이 원래 의도(각 필드 독립 LIKE)로 보임 |
| `.pc:332` | `sndl += (ii-1)*sizeof(orec[0])` | `ii==0`(자료없음) 시 `(0-1)`=-1 로 orec 1건 크기를 감산. `W1251S01_O_SZ` 가 orec[1] 을 포함하므로 결과적으로 헤더만 전송되어 우연히 정합. 의도가 산술에 의존하는 취약한 코드 |

confidence 참고: `.h`/`.pc` 두 파일 모두 존재하고 완독. 위 항목은 정적 판독 기준이며 런타임 미검증.
