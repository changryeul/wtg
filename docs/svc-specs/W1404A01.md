# 서비스 명세 — `W1404A01` (HOST 결제계좌 수집 및 등록)

> 작성 양식: `docs/svc-spec-template.md`. 본 명세는 `.pc` 실 구현 기준이며,
> `.h` 헤더 주석과 어긋나는 부분은 §6 주의에 정리했다.

## 1. 개요

| 항목 | 값 |
|---|---|
| 서비스 코드 | `W1404A01` |
| 이름 | HOST 결제계좌 수집 및 등록 (`.pc`/`.h` Description = "HOST결제계좌수집 및 등록") |
| 유형 | **복합 (외부 API 조회 → UPDATE/INSERT upsert)** — 코어뱅킹 e뱅킹 계좌조회 API(IF-054) 로 전계좌를 수집해 FX 결제계좌 원장에 신규 등록/갱신. `.pc` 함수 주석은 "조회" 이나 실제는 DML(수집·등록)임(§6) |
| 구현 | `win/src/trn/W1400/W1404A01.pc` (372 lines) |
| I/O 헤더 | `win/src/inc/trn/W1404A01.h` (+ 공통구조체 `CUS_ACNTS` = `win/src/inc/wws/WCM201.h`) |
| 라우팅 alias | `W1404A01` → `POST /v1/tx {"alias":"W1404A01","data":{…}}` |
| 채널 | 진입부에 `ctyp→chnl` 매핑 없음 (COMHDR 그대로 사용) |

## 2. I/O 전문 (`.h`)

**Input `W1404A01_I`** (총 17B)

| 필드 | 크기 | 설명 (실제) |
|---|---|---|
| `prGb` | 1 | 작업구분 — `'1'` 만 처리(수집·등록), 그 외 "작업구분 오류" 후 즉시 반환 |
| `wbcsRlnmAltrNo` | 16 | 전행고객실명대체번호(전행고객번호) — 계좌 수집 대상 키. 빈값이면 `32017` 로 거부 |

**Output `W1404A01_O`** (총 6B) — grid/orec 없음, 단일 필드

| 필드 | 크기 | 설명 |
|---|---|---|
| `grid01_cnt` | 6 | 헤더 주석은 "그리드건수" 이나 실제 처리건수 반환 X. 성공 시 `zredit(...,1)` 로 **항상 `1` 고정** (§6). grid record 자체가 없음 |

> 처리 결과 건수/목록은 전문으로 돌려주지 않는다. 실제 계좌 수집·등록 건수(`acnCnt`)는 로그로만 남고, 성공/실패는 COMHDR(`eflg`/`mesg`) 로 전달.

## 3. 구현 흐름 (`.pc`)

```
W1404A01()                       ← 진입 (콜백)
  ├─ prthdr / getmid / getmod    COMHDR + input(mid) + output(mod) 준비
  ├─ E_MSG 기본설정               ltyp="KR", mtyp="S"(FXON), rtyp="M"(Message전송)
  ├─ switch(mid->prGb)
  │     '1' → retc = W1404A01_F1(...)     ← 실제 로직
  │     그외 → comhdr_o->rtyp="M", mesg="작업구분 오류." 후 즉시 return
  │            (eflg 미설정, MSG_NOWORK 로그)
  ├─ retc==RET_SUCCESS ? eflg="0"+l_dbcommit() : eflg="1"+l_dbrollback()
  │                              (명시적 커밋/롤백, AUTOCOMMIT 아님)
  └─ fnSetMsg(&e_msg, comhdr_o) → SEND

W1404A01_F1()                    ← 수집·등록 (함수 주석은 "조회")
  ├─ getmid/getmod, memset(mod)
  ├─ 입력 바인딩: yprintfrtrim → vWbcsRlnmAltrNo, comhdr_i->usid → hls_upd_id
  │     strlen(vWbcsRlnmAltrNo)<=0 → e_msg "32017", RET_FAILURE
  ├─ 조회가능시간 체크
  │     SELECT CASE ... END INTO :vSchYn
  │       FROM TB_FXB_CMC003D
  │       WHERE COM_CD_NM='TRN_CNTL_TIM' AND CMCD_VL='0'
  │     SQL err → e_msg "10118", RET_FAILURE
  │     vSchYn=='N' (일자전환 거래제한) → e_msg "32713", RET_FAILURE  (조건식 주의 §6)
  ├─ cusAcnts 초기화, vWbcsRlnmAltrNo 세팅
  └─ for(;;)  연속데이터(페이징) 루프
       ├─ fnGetHostAcnts(&cusAcnts, &e_msg)   ← 코어뱅킹 e뱅킹 계좌조회 API(IF-054)
       │     (WCM201.pc: TB_FXB_CSC001M 로 RPRS_CUS_NO 조회 → apireq SVDE9212P1 HTTP 호출)
       │     실패 → RET_FAILURE (e_msg 는 함수 내부 set: 10165/43060/88888 등)
       ├─ for i in 0..cAcntCnt-1
       │     ├─ list[i] 필드 추출/rtrim (계좌/개설일/통화/상태/과목/상품)
       │     ├─ 통화 KRW 아니면 vCrcd="" + vFxSlactDcd='2'(외화), else '1'(원화)
       │     ├─ UPDATE TB_FXB_CSC003M  (개설일 최소값 유지 + 통화/상태/과목/상품/변경ID·TS)
       │     │     SQL err(≠NO_DATA) → e_msg "10122", RET_FAILURE
       │     ├─ NO_DATA_FOUND → INSERT TB_FXB_CSC003M (FX_SLACT_YN='N' 고정)
       │     │     SQL err → e_msg "10174", RET_FAILURE
       │     └─ else SQLCODE≠0 → e_msg "10175", RET_FAILURE
       │     acnCnt++
       └─ cExyYn[0]!='1' (다음 데이터 없음) → break
  ├─ acnCnt>0 → e_msg "10176" / else "10179"   (직후 91003 으로 덮임 §6)
  └─ zredit(mod->grid01_cnt,1), e_msg "91003"(조회 완료), RET_SUCCESS
```

## 4. 사용 DB 테이블

| 테이블 | 한글명 | 별칭 | 접근 | 용도 |
|---|---|---|---|---|
| `TB_FXB_CMC003D` | 코드정보상세 | — | **SELECT** | `COM_CD_NM='TRN_CNTL_TIM' AND CMCD_VL='0'` 로 거래통제시간(`CMCD_VL_REF_CON1/2`) 조회 → 조회가능시간 판정 |
| `TB_FXB_CSC003M` | 고객(FX)결제계좌 | — | **UPDATE / INSERT** | 수집된 계좌마다 `WBCS_RLNM_ALTR_NO + FX_SLACT_DCD + FX_STLM_ACN` 키로 UPDATE, 없으면(NO_DATA) INSERT (upsert) |
| `TB_FXB_CSC001M` | 고객약정정보기본 | — | **SELECT** (간접) | `fnGetHostAcnts()`(WCM201.pc) 내부에서 `WBCS_RLNM_ALTR_NO` 로 대표고객번호(`RPRS_CUS_NO`) 조회 → 코어뱅킹 API 입력에 사용 |

*DML 요약: 직접 SELECT 1문(`TB_FXB_CMC003D`) + 계좌 건별 UPDATE/INSERT(`TB_FXB_CSC003M`) + 공통함수 경유 SELECT 1문(`TB_FXB_CSC001M`). 계좌 실데이터는 DB 조회가 아니라 외부 코어뱅킹 HTTP API(`apireq`, svc_cd `SVDE9212P1`, IF-054)로 수집. 커서(DECLARE/OPEN/FETCH) 없음 — 페이징은 API `cExyYn`/`cLsInqAcno` 로 반복.*

## 5. 로직 / 필터 규칙

- **작업구분**: `prGb='1'` 만 유효. 그 외는 COMHDR `mesg="작업구분 오류."` 후 즉시 반환(eflg 미세팅).
- **입력 유효성**: `wbcsRlnmAltrNo` 빈값 거부(`32017`). 그 외 실질 검증 없음.
- **조회가능시간 게이트**: `TB_FXB_CMC003D`(`TRN_CNTL_TIM`) 의 시간대와 현재시각(`SYSDATE HH24MISS`) 비교로 `vSchYn` 산출, `'N'` 이면 "은행 일자전환 작업으로 거래 제한"(`32713`). ※ CASE 조건이 `(CON1<=now) OR (CON2>=now)` 로 되어 있어 사실상 대부분 `'N'` 이 나온다 — 로직 의심(§6).
- **계좌 수집 API**: `fnGetHostAcnts()` 가 `TB_FXB_CSC001M` 에서 대표고객번호를 얻어 코어뱅킹 e뱅킹 계좌조회 API(`SVDE9212P1`/IF-054)를 `apireq` 로 호출. 응답 `io_gi_row_cnt` 만큼 `list[]` 채움. 계좌상태는 API `dpz_ac_stsc=="01"` 이면 정상('1'), 아니면 해지('9').
- **페이징(연속 조회)**: API 응답 `cExyYn=='1'` 이면 다음 페이지 존재 → `cLsInqAcno`(최종조회계좌) 를 넘겨 루프 재호출. `'1'` 아니면 종료.
- **통화/구분 처리**: 통화가 `KRW` 가 아니면 `vCrcd` 를 빈값으로 만들고 `vFxSlactDcd='2'`(외화), `KRW` 이면 `'1'`(원화).
- **Upsert 규칙**: `TB_FXB_CSC003M` UPDATE 시 `ACNT_OPN_YMD` 는 `NVL(기존,'99991231') > 신규` 일 때만 신규값으로 갱신(최소 개설일 유지). UPDATE 가 `NO_DATA_FOUND` 면 INSERT, 이때 `FX_SLACT_YN='N'`(수집 계좌는 결제계좌 아님, 로그인 시 별도 등록) 고정. `FX_SYS_LSMD_ID`=사용자(usid), `FX_SYS_LSMD_TS`=SYSDATE.
- **트랜잭션**: `RET_SUCCESS→eflg="0"+l_dbcommit / 실패→eflg="1"+l_dbrollback`. AUTOCOMMIT 아님. 계좌 건별 실패 시 전체 롤백.
- **에러/결과 코드**:
  - 성공: `91003`("조회가 되었습니다."), `eflg=0`.
  - 고객번호 미입력: `32017`.
  - 조회가능시간 조회 SQL 오류: `10118`.
  - 거래제한 시간대: `32713`.
  - 계좌 API 실패: `fnGetHostAcnts` 내부 `10165`(대표고객번호 조회오류)/`43060`(API CALL 오류)/`88888`(뱅킹 업무오류) 등.
  - UPDATE 오류: `10122`, INSERT 오류: `10174`, UPDATE 후처리 오류: `10175`.
  - `10176`(수집 완료)/`10179`(등록계좌 없음) 는 세팅되나 직후 `91003` 으로 덮여 클라이언트엔 미도달(§6).
- **결과 전달**: `grid01_cnt`(항상 "1") + COMHDR(`eflg`/`e_msg.code`→`mesg`). WTG `/v1/tx` 는 `errn`≠0 이어도 전문 있으면 200, COMHDR 본문 동봉(레거시 규약).

> 성공 응답(91003)이 계좌 0건에도 나온다는 점은 UI/클라이언트 표기 설계 시 유의.
